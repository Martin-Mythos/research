# Harness Engineering 完整教程（面向工程与AI评测团队）

## 0. 教程目标与读者

**你将学会：**
1. 什么是 Harness Engineering（测试/评测“夹具与执行系统”工程化）。
2. 如何把“零散脚本”升级为可复现、可扩展、可审计的 Harness。
3. 如何在软件测试与 LLM/Eval 两类场景统一设计 Harness 架构。
4. 如何落地一个可直接运行的最小项目模板。

**目标读者：**
- 测试工程师、平台工程师、AI 应用评测工程师。
- 对自动化测试熟悉，但缺少“系统化 Harness 设计方法”的团队。

---

## 1. Harness Engineering 是什么？

Harness Engineering 指“把测试/评测执行体系工程化”的实践：
- 将 **测试对象（SUT）**、**输入数据集**、**执行器**、**判分器**、**结果存档**、**回归比较** 组织成可复用系统。
- 核心价值是 **可复现（Reproducibility）**、**可比较（Comparability）**、**可追溯（Traceability）**。

在传统测试里，它接近 test harness；在 AI 评测里，它扩展为 eval harness（例如 batch eval、prompt regression、模型对比）。

---

## 2. 问题驱动：为什么团队需要它？

常见痛点：
1. **脚本漂移**：每个人本地脚本不同，结果不可复现。
2. **数据污染**：测试集版本不清晰，导致指标波动无法解释。
3. **评分不一致**：人工判断口径不统一，回归结论争议大。
4. **不可审计**：上线前无法回答“这次结果和上次差异来自哪”。

Harness Engineering 的目标不是“多写脚本”，而是建立一个 **标准执行协议**。

## 3. 统一架构：七层 Harness 模型

### 3.1 配置层（Config）
- 用声明式配置定义：任务、数据版本、模型/服务版本、阈值、并发与重试。
- 所有实验都必须可由单一配置重放。

### 3.2 数据层（Data）
- 数据集分层：golden set（小而稳）+ regression set（覆盖变更）+ stress set（极限场景）。
- 强制版本化（hash/tag/date），禁止“隐式更新”。

### 3.3 执行层（Runner）
- 负责任务调度、并发、重试、超时、熔断。
- 执行日志必须结构化（JSONL/Parquet）。

### 3.4 评判层（Evaluator/Judge）
- 规则判分、模型判分、人工复核三者可组合。
- 对于主观任务，建议“自动评分 + 抽样人工仲裁”。

### 3.5 指标层（Metrics）
- 指标最少包含：质量、成本、时延、稳定性。
- 每个指标都要有“解释路径”（从聚合值追到单样本）。

### 3.6 比较层（Diff/Regression）
- 以 baseline 为锚点输出 delta（例如通过率变化、成本变化、P95 时延变化）。
- 定义“可接受波动区间”避免噪音触发错误告警。

### 3.7 治理层（Governance）
- 权限与审批：谁可以改数据、改阈值、改评分器。
- 产物留存：输入、输出、配置、代码版本、环境指纹完整保留。

---

## 4. 实施蓝图：从 0 到 1 落地

### 阶段 A：建立最小可运行 Harness（1~2 周）
- 固定 20~50 条黄金样本。
- 统一 CLI：`run -> eval -> report`。
- 输出一份基线报告（质量/成本/时延）。

### 阶段 B：接入回归与 CI（2~4 周）
- 每次 PR 触发子集回归。
- 主干每日跑全量回归。
- 结果自动归档并可视化。

### 阶段 C：治理与扩展（持续）
- 加入评分一致性监控。
- 数据变更走评审。
- 引入实验追踪系统（如 MLflow/W&B/自建元数据仓）。

## 5. 最小项目模板（可直接复制）

```text
harness-project/
  configs/
    baseline.yaml
  data/
    golden.jsonl
  harness/
    runner.py
    evaluators.py
    metrics.py
    report.py
  results/
  Makefile
  README.md
```

### 关键命令约定
- `make run CONFIG=configs/baseline.yaml`
- `make eval RUN_ID=...`
- `make report RUN_ID=...`
- `make diff BASE=run_a CAND=run_b`

这套约定比“脚本散落在 wiki”更重要，因为它定义了团队协作接口。

## 6. LLM 场景的 Harness 特别注意事项

1. **非确定性**：同一 prompt 结果会抖动。建议固定 temperature 或多次采样取分布统计。
2. **评测泄漏**：避免模型见过评测集；评测集要分层隔离。
3. **Judge 偏差**：LLM-as-a-judge 需定期校准并抽样人工对照。
4. **成本约束**：把 token 成本纳入主指标，而不是事后记账。
5. **安全合规**：PII 与敏感样本必须脱敏，并记录处理链路。

## 7. 质量门禁（Quality Gate）设计示例

一次候选版本通过门禁的示例条件：
- 质量：主指标下降不超过 1%。
- 成本：单请求成本上升不超过 5%。
- 时延：P95 不超过基线 + 200ms。
- 稳定性：失败率不高于 0.5%。

门禁不应是“单指标独裁”，而应是多目标平衡。

---

## 8. 常见反模式与修复建议

1. **只有平均分，没有分布** -> 增加 P50/P95/P99 与分桶统计。
2. **数据集长期不更新** -> 建立月度增量样本机制。
3. **把评测代码写进业务服务** -> 解耦为独立 harness 包。
4. **报告只看通过率** -> 增加失败案例 Top-K 与错误类型聚类。
5. **没有基线冻结** -> 给每次发布创建不可变 baseline。

---

## 9. 实战清单（Checklist）

上线前请确认：
- [ ] 任意结果可用同一配置重放。
- [ ] 数据集有版本号与变更记录。
- [ ] 评分器有一致性检查与抽样复核。
- [ ] 报告包含质量、成本、时延、稳定性四象限。
- [ ] 回归比较可追到样本级差异。
- [ ] 关键决策（放量/回滚）有证据链接。

---

## 10. 结语

Harness Engineering 的本质，是把“评测结论”从“经验判断”升级为“工程证据”。
当你的团队可以稳定回答“为什么这次好/坏、能否复现、是否值得上线”，Harness 就真正建成了。
